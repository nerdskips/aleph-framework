# Aleph Framework

Config-driven framework for WhatsApp AI agents. Define your agent entirely in YAML — no changes to `core/` needed, ever.

**Stack:** Python 3.12 · OpenAI Agents SDK 0.12 · FastAPI · Redis · Postgres/pgvector · Z-API (WhatsApp) · Bifrost (LLM router)

---

## Fast use

From zero to a running agent:

```bash
# 1. Install (no clone needed)
pip install "aleph-agent[all] @ git+https://github.com/nerdskips/aleph-framework"

# 2. Register the MCP server in Claude Code (once per machine)
claude mcp add --scope user aleph aleph-mcp

# 3. Create a folder for your agents and go in
mkdir my-agents && cd my-agents

# 4. Create your agent
aleph-agent init my-agent

# 5. Fill in credentials
#    Open my-agent/.env and set:
#      CLIENT_ID=my-agent
#      REDIS_URL=redis://...
#      OPENAI_API_KEY=sk-...  (or GEMINI_API_KEY, BIFROST_URL, etc.)

# 6. Write your agent's personality
#    Edit my-agent/prompts/system.md

# 7. Validate everything
aleph-agent test my-agent

# 8. Chat locally — no WhatsApp needed
aleph-agent chat my-agent

# 9. Deploy to Docker
aleph-agent start my-agent
```

**Useful after deploy:**
```bash
docker logs -f aleph-my-agent       # follow logs
curl http://localhost:8000/health    # health check
aleph-agent stop my-agent           # stop container
```

**With Claude Code (after step 2, no terminal needed):**
> "Create an agent for my bakery that answers product questions and escalates complaints"

Claude calls the MCP tools, creates and configures the agent, and tests it — you just review.

## How it works

Each agent lives in a folder under `clients/`:

```
clients/
  my-agent/
    config.yaml        # Full agent definition — this is the only file you need to edit
    prompts/
      system.md        # System prompt (supports {{placeholder}} injection)
    tools/
      my_tools.py      # Optional Python tools (@function_tool decorated)
    data/
      catalog.json     # Optional data files injected into the system prompt
    .env               # API keys and secrets — never committed
    Dockerfile         # Generated by aleph init, rarely needs editing
```

The framework reads `config.yaml`, builds the agent, wires guardrails, flows, tools, and knowledge base, then exposes a FastAPI webhook server that receives messages from Z-API (WhatsApp gateway).

## Environment variables

Every agent gets its own `.env` file at `clients/<name>/.env`. The framework loads it automatically on startup.

```bash
# Required
CLIENT_ID=my-agent            # Must match client_id in config.yaml

# LLM — pick ONE provider
BIFROST_URL=http://localhost:8080/v1   # Self-hosted Bifrost gateway (recommended)
BIFROST_API_KEY=dummy

# Or connect directly to a provider:
# OPENAI_API_KEY=sk-proj-...
# GEMINI_API_KEY=AIzaSy-...
# DEEPSEEK_API_KEY=sk-...
# OPENROUTER_API_KEY=sk-or-...

# Redis (required)
REDIS_URL=redis://:password@host:6379/0

# Z-API / WhatsApp (required for production webhook)
ZAPI_INSTANCE=your-instance-id
ZAPI_TOKEN=your-token
ZAPI_CLIENT_TOKEN=your-client-token

# Database (only if habits.enabled=true or knowledge.enabled=true)
DATABASE_URL=postgresql://user:pass@host:5432/dbname

# Suppress SDK tracing noise in development
OPENAI_AGENTS_DISABLE_TRACING=1
```

## MCP Server (Claude Code integration)

Register once — available in every project on this machine:

```bash
pip install aleph-agent          # or: pip install -e ".[dev,all]" in dev mode
claude mcp add --scope user aleph aleph-mcp \
  --env ALEPH_CLIENTS_DIR=/absolute/path/to/your/clients
```

> **Why `ALEPH_CLIENTS_DIR`?** When installed via `pip` or `uv tool install`, the
> `aleph-mcp` binary runs from inside the package's virtual environment — it has no
> way to know where your agents live. Set this once and every MCP tool will find them.

Verify:

```bash
claude mcp list
# aleph    stdio    aleph-mcp
```

Claude Code can now call these tools directly in conversation:

| Tool | What it does |
|---|---|
| `list_agents` | List all agents with status |
| `create_agent(name, ...)` | Scaffold a new agent from templates |
| `validate_agent(name)` | Run config validation, return PASSED/FAILED |
| `get_config(name)` | Read `config.yaml` |
| `update_config(name, content)` | Write `config.yaml` (validates before saving) |
| `get_system_prompt(name)` | Read `prompts/system.md` |
| `update_system_prompt(name, content)` | Write `prompts/system.md` |
| `chat_message(name, message)` | Send one message through the full pipeline |

## CLI commands

All commands are run from the `clients/` directory (or the directory containing your agent folders).

```bash
cd clients/
```

### `aleph-agent init <name>`

Creates a new agent scaffold with all required files.

```bash
aleph-agent init my-agent
aleph-agent init my-agent --port 8099 --model openai/gpt-4.1-mini
```

Creates:
```
my-agent/
  config.yaml
  prompts/system.md
  tools/.gitkeep
  data/.gitkeep
  .env
  Dockerfile
  .dockerignore
  .gitignore
```

After init, edit `.env` with credentials and `prompts/system.md` with your agent's personality, then run `aleph-agent test my-agent`.

---

### `aleph-agent test <name>`

Validates the agent without starting a server or needing any credentials. Checks:

1. YAML syntax
2. Schema validation against all Pydantic models
3. Required files (system prompt, .env, Dockerfile)
4. Tools (webhook URLs present, code modules exist)

```bash
aleph-agent test my-agent
```

Exit code 0 = ready to deploy. Exit code 1 = fix errors first.

---

### `aleph-agent chat <name>`

Interactive chat in the terminal. No WhatsApp connection needed. Full pipeline runs: guardrails, flows, tools, knowledge search — everything except actual message sending.

```bash
cd clients/
aleph-agent chat my-agent
aleph-agent chat my-agent --log-level DEBUG
```

The chat loop shows guardrail decisions, tool calls, LLM timing, and flow state transitions inline.

**Note:** Flows require Redis (set `REDIS_URL` in `.env`). Chat mode connects to the same Redis instance used in production.

---

### `aleph-agent start <name>`

Builds the Docker image and starts the container.

```bash
aleph-agent start my-agent
aleph-agent start my-agent --port 8099   # override port
aleph-agent start my-agent --no-build    # skip image build
```

The container is named `aleph-<name>` and uses `--restart unless-stopped`. If a Docker network named `zuper_net` exists, the container joins it automatically.

After start:
```bash
docker logs -f aleph-my-agent   # follow logs
curl http://localhost:8099/health
```

---

### `aleph-agent stop <name>`

Stops and removes the container.

```bash
aleph-agent stop my-agent
```

---

### `aleph-agent list`

Lists all agent directories in the current folder and shows which are running.

```bash
cd clients/
aleph-agent list
```

---

### `aleph-agent knowledge <action> <name>`

Manages the RAG knowledge base. Requires `knowledge.enabled: true` in `config.yaml` and `DATABASE_URL` in `.env`.

```bash
# Load a single file
aleph-agent knowledge load my-agent --file data/manual.pdf

# Load all files in a directory
aleph-agent knowledge load my-agent --dir data/docs/

# List what's in the knowledge base
aleph-agent knowledge list my-agent

# Clear all knowledge for this agent
aleph-agent knowledge clear my-agent

# Clear by source file
aleph-agent knowledge clear my-agent --source manual.pdf
```

Supported formats: `.pdf`, `.md`, `.txt`, `.json`, `.csv`

## YAML reference

```yaml
client_id: "my-agent"    # required — must match CLIENT_ID in .env
```

### `agent`

Core agent identity. The minimum required section.

```yaml
agent:
  name: "My Agent"                        # required — display name
  description: "What this agent does"     # shown in tracing/logs
  model: "openai/gpt-4.1-mini"            # primary LLM via Bifrost
  fallback_model: "google/gemini-2.5-flash"  # used if primary fails
  temperature: 0.7                         # 0.0–2.0
  max_tokens: 1024
  system_prompt_file: "prompts/system.md"  # relative to agent dir
  parallel_tool_calls: true                # DEFAULT ON — concurrent tool execution (Phase 10)
```

**System prompt** supports `{{key}}` placeholders for `data_files` injection. Example: if you have a `data_files` entry with `key: products`, write `{{products}}` in the prompt and the file contents are injected at boot.

**Model identifiers** follow the Bifrost format: `provider/model-name`. Examples: `openai/gpt-4.1-mini`, `openai/gpt-4.1`, `google/gemini-2.5-flash`, `deepseek/deepseek-chat`.

### `guardrails`

Deterministic pre-LLM and post-LLM rules. Run before the LLM (input) and after (output). Zero LLM cost.

#### Input patterns

```yaml
guardrails:
  input_patterns:

    # REDIRECT — return a static reply, skip LLM entirely
    - name: "saudacao"
      keywords: ["oi", "olá", "bom dia", "boa tarde"]
      action: redirect
      redirect_message: "Olá! Como posso ajudar?"
      priority: 10

    # INJECT — add a hidden instruction to the LLM input, then call LLM
    - name: "preco_inject"
      keywords: ["preço", "valor", "quanto custa"]
      action: inject
      inject_instruction: "O usuário perguntou sobre preço. Use apenas os dados do catálogo."
      priority: 5

    # BLOCK — return a safe "I can't help with that" response
    - name: "concorrente"
      keywords: ["concorrente", "outra empresa"]
      action: block
      priority: 8

    # ESCALATE — pause bot and notify a human (checks habits first if enabled)
    - name: "reclamacao"
      keywords: ["reclamar", "fui lesado", "quero falar com humano"]
      action: escalate
      priority: 9

    # TOOL_REQUIRED — force the LLM to call a specific tool
    - name: "busca_cep"
      regex: ["\\b\\d{5}-?\\d{3}\\b"]
      action: tool_required
      tool_choice: "buscar_cep"
      priority: 7
```

**All input actions:**

| Action | LLM called? | Effect |
|---|---|---|
| `redirect` | No | Returns `redirect_message` directly |
| `block` | No | Returns a safe "can't help" response |
| `inject` | Yes | Adds `inject_instruction` to system prompt |
| `escalate` | Maybe | Checks habits; escalates to human if no match |
| `escalate_no_habit` | No | Always escalates, never checks habits |
| `takeover` | No | Human takes over the full conversation |
| `tool_required` | Yes | Forces `tool_choice` to the specified tool |
| `continue` | Yes | Passes through normally (explicit no-op rule) |

**Pattern matching** — keywords are matched against the normalized message (lowercase, no accents). Regex patterns follow Python `re` syntax. Both can be combined in the same rule. Higher `priority` wins when multiple patterns match.

#### Output guardrails (always on by default)

```yaml
guardrails:
  enable_fabrication_guard: true      # blocks invented addresses, fake data
  enable_price_leak_guard: true       # blocks loose prices without context
  enable_ghost_escalation_guard: true # catches LLM claiming to escalate without calling the tool

  output_rules:
    - name: "no_cpf"
      type: custom_regex
      patterns: ["\\d{3}\\.\\d{3}\\.\\d{3}-\\d{2}"]
      safe_response: "Não posso compartilhar dados pessoais."
```

### `flows`

Declarative multi-step conversation flows. The framework controls the conversation state; the LLM only runs when needed. State is stored in Redis at `aleph:{client_id}:flow:{phone}`.

**Default: OFF** — enable only when the agent needs to guide users through a structured process.

```yaml
flows:
  enabled: true
  default_state_ttl: 1800      # seconds of inactivity before state expires (default: 30min)
  flows:

    # Simple onboarding — collect name and interest, then hand off to LLM
    - id: onboarding
      trigger_keywords: ["começar", "quero contratar", "onboarding"]
      trigger_regex: []           # optional — Python regex patterns
      on_interrupt: pause         # hold | pause (default: pause)
      steps:
        - id: ask_name
          message: "Qual é o seu nome?"
          collect_as: nome        # stored as state.collected["nome"]
          next: ask_interest      # ID of the next step
        - id: ask_interest
          message: "Qual produto te interessa, {nome}?"  # {key} resolved from collected
          collect_as: interesse
          # no 'next' = last step — triggers on_complete
          on_complete:
            action: continue_to_llm    # inject collected data and pass to LLM
            inject_summary: true        # prepends collected data to user message

    # Checkout — collect address, call a webhook, confirm to user
    - id: checkout
      trigger_keywords: ["finalizar pedido", "comprar"]
      state_ttl: 600             # override default TTL for this flow (10min)
      on_interrupt: hold         # ignore off-topic messages, re-ask the step
      steps:
        - id: ask_address
          message: "Confirme seu endereço de entrega:"
          collect_as: endereco
          on_complete:
            action: webhook
            url: "http://n8n:5678/webhook/checkout"
            method: POST
            then: send_message   # after webhook returns, send the message below
            message: "Pedido registrado! Você receberá a confirmação em breve."

    # Qualification — collect budget, escalate to human with context
    - id: qualification
      trigger_keywords: ["tenho interesse", "quero saber mais"]
      steps:
        - id: ask_budget
          message: "Qual é o seu orçamento mensal?"
          collect_as: orcamento
          on_complete:
            action: escalate    # sends collected data to human with context
```

**`on_interrupt`** — controls what happens when the user goes off-topic mid-flow:
- `pause` (default): LLM answers the off-topic message, then the step question is appended automatically.
- `hold`: the off-topic message is ignored, the step question is re-sent, LLM is skipped.

Off-topic detection is deterministic: a message matches another flow's `trigger_keywords` or `trigger_regex`.

**`on_complete` actions:**

| Action | Effect |
|---|---|
| `send_message` | Sends the static `message` field, flow ends |
| `continue_to_llm` | Injects collected data into LLM context, continues conversation |
| `webhook` | POSTs collected data to `url`; optionally sends `message` after (`then: send_message`) |
| `escalate` | Escalates to human with collected data as context |
| `start_flow` | Chains into another flow (set `flow_id`) |

### `tools`

Register tools the LLM can call. Two types: `webhook` (pure YAML, no code) and `code` (Python).

#### Webhook tool

Zero code required. The framework generates a `@function_tool` automatically.

```yaml
tools:
  # GET with URL template (REST API pattern)
  - name: "buscar_cep"
    type: webhook
    description: "Busca endereço completo por CEP brasileiro"
    webhook_url: "https://viacep.com.br/ws/{cep}/json/"
    method: GET
    parameters:
      cep:
        type: string
        description: "CEP (somente números, 8 dígitos)"
        required: true

  # POST to N8N (body params — default for POST)
  - name: "criar_lead"
    type: webhook
    description: "Cria um lead no CRM"
    webhook_url: "http://n8n:5678/webhook/lead"
    method: POST
    parameters:
      nome:
        type: string
        description: "Nome do cliente"
      telefone:
        type: string
        description: "Telefone do cliente"
      acao:
        type: string
        default: "criar_lead"
        hidden: true          # injected automatically, not visible to LLM

  # GET with query parameters
  - name: "buscar_produto"
    type: webhook
    description: "Busca produto no catálogo"
    webhook_url: "https://api.example.com/produtos"
    method: GET
    params_in: query           # appended as ?nome=...&categoria=...
    parameters:
      nome:
        type: string
        description: "Nome do produto"
```

**`params_in` options:** `auto` (default — detects from URL and method), `url` (resolves `{param}` in URL), `body` (JSON POST body), `query` (URL query string).

#### Code tool

Python function in the agent's `tools/` directory. Decorate with `@function_tool` from the OpenAI Agents SDK.

```yaml
tools:
  - name: "minha_ferramenta"
    type: code
    module: "my_tools"           # loads tools/my_tools.py
    # functions: ["func_a"]      # optional — auto-discovers @function_tool if omitted
```

```python
# tools/my_tools.py
from __future__ import annotations
from agents import function_tool

@function_tool
def minha_ferramenta(param: str) -> str:
    """What this tool does — shown to the LLM."""
    return f"Result: {param}"
```

### `human`

Human-in-the-loop — escalation and takeover. **Default: ON.**

```yaml
human:
  enabled: true
  responsible_phones:                # notified when escalation triggers
    - "5511999990000"
    - "5511999990001"
  escalation_session_ttl: 7200       # seconds the bot stays paused (default: 2h)
  takeover_lock_ttl: 3600            # seconds for full takeover lock (default: 1h)
  takeover_renew_on_message: true    # each human message renews the lock
  release_keyword: "RELEASE"         # human types this to hand back to bot
  notify_via: "zapi"                 # zapi | n8n
```

**Escalation flow:**
1. Input guardrail `action: escalate` triggers → bot sends holding message to user → responsible phones are notified with the user's message.
2. Human replies by quoting the notification → reply is forwarded to the user.
3. Bot remains paused until the lock expires or human types `RELEASE`.

### `habits`

Operational memory — learns from human resolutions and uses them to answer future similar questions without escalating. **Default: OFF.**

Requires `DATABASE_URL` in `.env` and a Postgres database with the `pgvector` extension.

```yaml
habits:
  enabled: true
  auto_migrate: true           # creates table/indexes on first boot
  embedding_model: "openai/text-embedding-3-small"
  dedup_threshold: 0.020       # cosine distance — lower = stricter dedup
  match_count: 3               # max results from hybrid search
  search_before_escalate: true # check habits before escalating to human
  table_name: "operational_habits"
  schema: "public"             # use a different schema for Supabase isolation
```

### `knowledge`

RAG knowledge base using hybrid search (BM25 + vector + RRF reranking). **Default: OFF.**

Requires `DATABASE_URL` in `.env` and a Postgres database with the `pgvector` extension.

```yaml
knowledge:
  enabled: true
  auto_migrate: true
  embedding_model: "openai/text-embedding-3-small"
  schema: "knowledge"
  table_name: "knowledge_base"
  auto_search: true            # search before every LLM call
  auto_search_top_k: 5         # chunks to inject into LLM context
  tool_search: true            # also expose as a callable tool
  similarity_threshold: 0.7
  chunk_size: 500              # tokens per chunk during ingestion
  chunk_overlap: 75
```

Load files with `aleph-agent knowledge load <name> --file path/to/file.pdf`.

### `follow_up`

```yaml
    level: INFO              # DEBUG | INFO | WARNING | ERROR
    format: json             # json | text
    log_to_file: ""          # optional log file path

  tracing:
    enabled: true
    export_to: console       # console | braintrust | opentelemetry | custom
    sample_rate: 1.0         # 1.0 = trace everything

  verbose_errors: true       # include stack traces in error responses (disable in prod)
  dry_run: false             # true = full pipeline but no messages sent (testing)
```

### `subagents`

Specialist sub-agents invoked as tools by the main agent (MANAGER pattern). **Default: OFF.** The orchestrator LLM decides when to call each sub-agent — they run their own Agent+Runner loop and return a result. Combine with `parallel_tool_calls: true` to invoke multiple sub-agents concurrently.

```yaml
subagents:
  - name: logistics                          # internal agent name
    tool_name: consultar_entrega             # tool name the LLM calls
    tool_description: "Consulta prazos e status de entrega para um CEP e produto"
    instructions: "Você é especialista em logística. Responda apenas sobre prazos e fretes."
    model: openai/gpt-4.1-nano              # optional — inherits main model if omitted
    max_turns: 5                             # max turns for this sub-agent's loop
    tools:                                   # optional — same syntax as main agent tools
      - name: frete_api
        type: webhook
        webhook_url: "${FRETE_API_URL}"
        parameters:
          cep: {type: string, description: "CEP de destino", required: true}

  # Alternative: ref mode — load instructions from another agent's system prompt
  - name: financeiro
    tool_name: consultar_financeiro
    tool_description: "Consulta boletos, faturas e situação financeira do cliente"
    ref: financeiro                          # points to clients/financeiro/ directory
```

### `queue`

Background job dispatcher — fire-and-forget HTTP webhooks after pipeline events. **Default: OFF.** The pipeline enqueues jobs without blocking the user's reply. A separate worker process (`run_worker`) consumes the queue.

```yaml
queue:
  enabled: true
  max_retries: 3
  jobs:
    - trigger: pipeline_complete             # fires after every successful response
      action: webhook
      webhook_url: "${CRM_WEBHOOK_URL}"
      include_fields: [phone, response, elapsed_seconds]
      timeout_seconds: 10

    - trigger: flow_complete                 # fires when a specific flow finishes
      flow_id: onboarding                    # only fires for this flow
      action: webhook
      webhook_url: "${ORDERS_WEBHOOK}"
      include_fields: [phone, collected]     # collected = flow's gathered data

    - trigger: escalation_start             # fires when human escalation begins
      action: webhook
      webhook_url: "${SLACK_WEBHOOK}"
      include_fields: [phone, last_message]
```

**Triggers:** `pipeline_complete` | `flow_complete` | `escalation_start`

### `self_awareness`

Prior state injection — injects interrupted flow context, active escalation state, and episodic memory summary when a user returns after a gap. **Default: OFF.**

```yaml
self_awareness:
  enabled: true
  return_gap_minutes: 30        # fire only when user returns after 30min+ gap
  max_injection_age_hours: 4    # ignore states older than 4h
  include_flow: true            # inject interrupted flow context
  include_escalation: true      # inject escalation state if active
  include_summary: true         # inject episodic memory summary
```

## Message pipeline

Every incoming WhatsApp message passes through this pipeline in order:

```
Z-API webhook
  ↓
1. Messaging filter       — discard groups, broadcasts, reactions, edits
2. Human reply detection  — route human HITL replies to the right conversation
3. Anti-spam              — deduplicate redelivered messages (Redis, 120s TTL)
4. Message buffer         — consolidate multi-chunk messages (8s window)
5. Processing lock        — per-phone lock prevents race conditions
   ↓
1.1 Human takeover check  — if human has taken over, hold all messages
1.2 Input guardrail       — deterministic keyword/regex rules (9 possible actions)
1.5 Escalation check      — if escalation active, hold with waiting message
1.6 Flow resolution       — state machine step advancement (if flows.enabled)
1.8 Knowledge search      — inject relevant chunks pre-LLM (if knowledge.enabled)
   ↓
2. Agent SDK run           — primary model → fallback model on failure
   ↓
3. Output guardrail        — fabrication, price leak, ghost escalation checks
   ↓
4. Humanized send          — split by paragraph, add delays, append disclaimer
```

## Architecture standards

Standards enforced across the entire `core/` codebase. Follow these when contributing new modules or extending existing ones.

### Module structure

Every `.py` file follows this import order:

```python
"""Aleph Framework — Module Name.
Brief description of what this module does.
"""

from __future__ import annotations   # always — enables PEP 563 postponed annotation evaluation

import os                            # stdlib
import logging

from pydantic import BaseModel       # third-party

from core.registry.schema import ... # internal
```

`from __future__ import annotations` is **mandatory** in every `.py` file, including `__init__.py` and test files.

### Logger naming

Always `logging.getLogger("aleph.module.submodule")` — mirrors the filesystem hierarchy:

```python
logger = logging.getLogger("aleph.engine.pipeline")   # core/engine/pipeline.py ✅
logger = logging.getLogger("aleph.queue.dispatcher")  # core/queue/dispatcher.py ✅
logger = logging.getLogger("aleph.pipeline")          # ❌ too shallow — wrong
```

### Config passing

Always pass typed config objects, never individual parameters:

```python
# ✅ full FrameworkConfig — for modules that need the whole config
def __init__(self, config: FrameworkConfig): ...

# ✅ feature-specific config — for modules that own a single feature
def __init__(self, config: HabitsConfig): ...

# ❌ individual params — breaks encapsulation
def __init__(self, model: str, temperature: float, max_tokens: int): ...
```

Store config at `self.config` and access nested values as `config.feature.field`.

### Redis key format

All keys follow `aleph:{client_id}:{feature}:{detail}` — the `client_id` segment provides multi-agent isolation on a shared Redis instance:

```
aleph:{client_id}:spam:{message_id}       # anti-spam dedup
aleph:{client_id}:lock:{phone}            # processing lock
aleph:{client_id}:buffer:{phone}          # message consolidation buffer
aleph:{client_id}:flow:{phone}            # flow state machine
aleph:{client_id}:queue                   # background job queue
aleph:{client_id}:escalation:{phone}      # escalation state
```

Lowercase, colon-separated — never dots, dashes, or underscores as separators.

### Async/sync split

I/O operations are `async def`; pure logic is `def`:

```python
async def connect(self) -> None: ...             # ✅ Redis/DB/HTTP — async
def classify_input(text, config) -> ...: ...     # ✅ pure logic — sync

async def bad(self):
    time.sleep(1)       # ❌ blocks the event loop
    requests.get(url)   # ❌ sync HTTP in async context
```

All test functions can be `async def` — `asyncio_mode = "auto"` in `pyproject.toml` handles it.

### Lazy imports

Use only for optional dependencies or to break circular imports:

```python
# ✅ optional dep — import inside the function
async def ingest_pdf(path: str) -> str:
    from pypdf import PdfReader
    ...

# ✅ feature-gated — import at the call site
if config.queue.enabled:
    from core.queue.dispatcher import dispatch_jobs

# ❌ regular dep — move to top of file
def process(text: str):
    import json   # wrong: stdlib should be at top
```

### Feature flags

All new features are **DEFAULT OFF** in the Pydantic schema:

```python
class NewFeatureConfig(BaseModel):
    enabled: bool = Field(False, description="DEFAULT OFF — enable via config.yaml")
```

Never enable a feature unconditionally — always gate on `config.feature.enabled`.

### Test conventions

Each test file covers one module. Use `_make_` builder functions instead of shared fixtures:

```python
"""Tests: FeatureName — scenarios covered."""

from __future__ import annotations

from unittest.mock import AsyncMock

def _make_config(**overrides) -> FrameworkConfig:
    base = {"client_id": "test", "agent": {"name": "Bot"}}
    base.update(overrides)
    return FrameworkConfig(**base)

def _mock_redis() -> AsyncMock:
    redis = AsyncMock()
    redis.get = AsyncMock(return_value=None)
    return redis

def test_feature_default_off():
    config = _make_config()
    assert config.feature.enabled is False

async def test_feature_async_operation():
    result = await some_async_call(_mock_redis())
    assert result.status == "ok"
```

Rules: no real Redis/DB in unit tests (`AsyncMock` only), one assertion per test, naming `test_{component}_{scenario}`.
